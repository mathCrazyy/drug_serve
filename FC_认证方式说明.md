# 函数计算 HTTP 触发器认证方式说明

## 📋 认证方式选项

在函数计算的 HTTP 触发器配置中，有 5 种认证方式：

1. **无需认证**（No authentication）✅ **选择这个**
2. 签名认证（Signature authentication）
3. Basic 认证（Basic authentication）
4. JWT 认证（JWT authentication）
5. Bearer 认证（Bearer authentication）

## ✅ 应该选择：无需认证

**"无需认证"就是匿名访问的意思**，适合前端直接调用。

### 为什么选择"无需认证"？

1. **前端需要直接访问**
   - 前端通过 HTTP 请求调用后端
   - 不需要复杂的认证流程

2. **简单易用**
   - 不需要配置额外的认证信息
   - 可以直接通过浏览器或 curl 测试

3. **适合本项目**
   - 我们的项目是前后端分离架构
   - 前端需要直接调用后端 API

## 🔧 如何修改

### 在触发器编辑页面

1. **找到"认证方式"部分**
   - 你会看到 5 个单选按钮
   - 当前选中的是"签名认证"

2. **选择"无需认证"**
   - 点击"无需认证"（第一个选项）
   - 单选按钮会变成蓝色，表示已选中

3. **保存修改**
   - 点击"确定"或"保存"按钮
   - 等待保存完成

## 📊 认证方式对比

| 认证方式 | 适用场景 | 配置复杂度 | 本项目 |
|---------|---------|-----------|--------|
| **无需认证** | 前端调用、公开 API | ⭐ 简单 | ✅ 推荐 |
| 签名认证 | 服务间调用 | ⭐⭐⭐ 复杂 | ❌ |
| Basic 认证 | 需要用户名密码 | ⭐⭐ 中等 | ❌ |
| JWT 认证 | 需要 Token | ⭐⭐ 中等 | ❌ |
| Bearer 认证 | 需要 Token | ⭐⭐ 中等 | ❌ |

## ⚠️ 注意事项

1. **"无需认证"的安全性**
   - 意味着任何人都可以调用你的 API
   - 对于这个项目，这是可以接受的
   - 如果担心安全，可以在后端代码中添加 API Key 验证

2. **修改后需要测试**
   - 修改认证方式后，立即测试
   - 使用 curl 命令验证是否可访问

## 🧪 修改后测试

修改为"无需认证"后，测试：

```bash
curl https://drug-se-backend-laubwkztsv.cn-hangzhou.fcapp.run/
```

应该返回：
```json
{"message":"药品识别与提醒系统 API","docs":"/docs"}
```

如果还是返回错误，请检查：
1. 是否已保存修改
2. 是否等待了几秒钟（配置生效需要时间）
3. 是否选择了正确的认证方式

---

**总结：选择"无需认证"，这就是匿名访问！** ✅
